## 总目标

在不扩张范围的前提下，把当前仓库推进到“一期可交付、可运营、可商用测试”的状态。

一期总目标如下：

- 足球、篮球核心页面的数据稳定可用。
- 板球具备可访问的 live 入口、基础比分展示与多语言文案。
- 会员套餐与单条内容购买形成真实交易闭环。
- 后台可完成内容发布、赛事同步、首页配置、用户与订单查询。
- 前台内容权限与会员权益逻辑统一、稳定、可回溯。
- PC 与移动 H5 具备上线前的基本稳定性。

## 开发原则

1. 先做数据基础，再做商业化扩展。
2. 先做支付与权限正确性，再做大规模运营能力。
3. 先把后台 CRUD 做完整，再让运营深度依赖系统。
4. 先把核心链路做稳，再补性能、SEO、监控和上线细节。

约束如下：

- 一期核心深度仍聚焦足球和篮球。
- 板球按分阶段交付执行：
  - 先完成 live 入口、赛程比分展示、多语言适配。
  - 后续再补资料库、内容、后台配置与更深数据。
- 不扩展到原生 App。
- 不在本阶段自建大型采集系统，优先把现有抓取链路做稳定。
- 每个阶段结束必须有验证记录，不能只停在“代码写完”。

## 阶段排期

### Week 1：业务数据模型正式化

目标：
把体育数据和内容数据从页面展示结构升级为可持久化的正式业务模型。

已完成：

- 扩展 Prisma schema，补齐 League、Team、Match、OddsSnapshot、PredictionRecord、ArticlePlan、AuthorTeam。
- 梳理 provider、mapper、repository 三层职责。
- 为后续同步入库和前台聚合读取打下基础。

验收：

- 新 schema 能覆盖首页、比分页、资料库页、计划单页的数据需求。
- 实体关系清晰，可追踪，可被 repository 层稳定消费。

### Week 2：同步入库链路

目标：
从“实时抓取即展示”切换到“同步入库 -> 页面读取”的稳定模式。

已完成：

- 建立联赛、球队、比赛、赔率、赛程、积分、历史交锋同步任务。
- `/api/admin/sync` 可触发真实同步并返回结果。
- 关键页面优先从本地数据库读取已同步数据。
- 对上游部分失败场景做了隔离，不会拖垮整次同步。

验收：

- 首页、足球直播、篮球直播、资料库、赛事详情可读取本地同步数据。
- 重复同步不会明显产生脏数据。

### Week 3：计划单与作者体系

目标：
把当前依赖 mock 的推荐内容体系替换为后台可维护、前台可展示的正式内容体系。

已完成：

- 后台支持作者/团队管理。
- 后台支持计划单内容 CRUD、发布切换、热门切换、归档。
- 前台 `/plans` 与 `/plans/[slug]` 已改为正式数据源。
- 单条购买链路已按数据库价格读取内容。

验收：

- 运营可在后台创建、编辑、发布计划单。
- 前台计划单列表和详情可实时反映后台数据。

### Week 4：会员权益与权限治理

目标：
把账号体系从“能用”提升到“权限统一、逻辑一致、可维护”。

当前完成：

- 统一角色模型：`visitor`、`member`、`operator`、`admin`
- 新增 entitlement 服务，集中判断：
  - 是否已登录
  - 会员是否有效
  - 是否可进入后台
  - 是否可查看某条完整内容
- 布局层统一下发 `session + entitlements`
- 会员中心、计划单列表、计划单详情全部切换到同一套解锁逻辑
- 后台页面与后台接口已切到统一权限判断
- 后台 `users` tab 已接入真实用户与订单数据

本周剩余：

- 本地手工 smoke test：
  - `/login`
  - `/member`
  - `/plans`
  - `/plans/[slug]`
  - `/admin`
- 根据手测结果微调 returnTo 流程和空状态细节

验收：

- 非会员不能直接查看完整付费内容。
- 单篇已购用户可查看对应全文。
- 有效会员可查看会员权益内内容。
- 非管理员无法进入后台核心页面。

### Week 5：真实支付与订单状态机

目标：
把当前直接写 `paid` 的演示购买链路，升级为真实可商用的交易流程。

任务：

- 为会员订单和内容订单引入状态机：
  - `pending`
  - `paid`
  - `failed`
  - `closed`
  - `refunded`
- 接入明确的 Web 支付通道。
- 建立下单、回调、签名校验、补单、异常关闭流程。
- 保存支付流水、回调摘要、失败原因、状态更新时间。
- 确保会员权益和内容解锁只在支付成功后生效。

验收：

- 未收到合法回调前，订单不能被标记为 `paid`。
- 重复回调不会重复生效。
- 支付超时、失败、取消有清晰状态和恢复路径。

### Week 6：后台运营完善

目标：
把后台从“有入口”升级为“日常可用的运营后台”。

任务：

- 首页模块配置：
  - Banner
  - 焦点赛事
  - 热门联赛
  - 推荐位
  - 作者榜单
- 用户列表、会员状态、订单查询增强。
- AI 导入记录和人工修正记录。
- 操作审计字段补齐：
  - 创建人
  - 更新人
  - 更新时间

验收：

- 运营无需改代码即可完成首页配置。
- 客服或运营可在后台查询用户和订单。
- AI 预测展示内容可导入并可追踪修正历史。

### Week 7：性能、缓存与 SEO

目标：
在核心功能稳定后，补齐上线级的基础性能和可发现性能力。

任务：

- 首页、直播页、资料库页建立缓存与 revalidate 策略。
- 降低重复抓取和重复查询压力。
- 增加基础 SEO：
  - metadata
  - canonical
  - sitemap
  - robots
- 补齐 loading、empty、error 状态。
- 检查 PC 与移动 H5 的字体、布局、加载表现。

验收：

- 上游短时异常时，主要页面仍可读。
- 公共页面具备基础搜索引擎元信息。
- 高频访问不会产生不必要抓取压力。

### Week 8：联调、验收与上线准备

目标：
完成一期上线前的最终联调、回归、问题收口和发布准备。

任务：

- 端到端回归：
  - 登录
  - 浏览比分/资料
  - 购买会员
  - 购买单条
  - 解锁内容
  - 后台发布
  - 数据同步
- PC 与移动 H5 的交互和布局复核。
- 错误日志、基础 tracing、发布检查清单。
- seed 数据、运营使用说明、回滚预案。
- 上线后问题跟踪清单。

验收：

- 用户链路可闭环完成。
- 运营链路可闭环完成。
- 已知问题有明确优先级和责任人。

## 跨周持续项

以下事项不能只留到最后一周，必须每周处理：

- 文案与 UI 一致性
- 空状态、错误状态、边界状态
- Provider 抓取异常兜底
- 核心接口与关键业务链路测试
- 关键转化页面的埋点准备

如果进度延期，优先级顺序如下：

1. 数据正确性
2. 支付与权限正确性
3. 后台运营可用性
4. 视觉打磨与次级优化

## 完成定义

一个功能只有满足下面条件，才算真正完成：

- 代码已落地并可本地验证
- 主链路至少完成一次 smoke test
- 空状态和错误状态已处理
- 用户可见文案清晰可读
- 涉及迁移、初始化或规则调整时有文档记录
